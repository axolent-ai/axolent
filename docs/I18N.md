# Internationalization (i18n)

AXOLENT AI supports 20 languages via a JSON-based internationalization system
with pre-commit enforcement. This document covers the system architecture,
how to add new keys, and the quality gates.

## Overview

* **20 languages** with full key parity
* **154+ translation keys** covering all user-facing bot responses
* **Source hash tracking** to detect stale translations when English text changes
* **AST scanner** to catch hardcoded strings in production code
* **English-only enforcement** in production code paths
* **Placeholder validation** to ensure translated strings use the same `{variables}`

## Locale File Structure

Locale files live in `bridge/i18n/locales/`:

```
bridge/i18n/locales/
    _meta.json      Language metadata (native name, english name, RTL flag)
    en.json         English (source of truth)
    de.json         German
    fr.json         French
    ...             (20 files total)
```

Each locale file is a flat JSON object:

```json
{
  "welcome": {
    "text": "Welcome to AXOLENT AI!",
    "source_hash": "a1b2c3d4e5f6g7h8"
  },
  "help_intro": {
    "text": "Here are the available commands:",
    "source_hash": "b2c3d4e5f6g7h8i9"
  }
}
```

**Fields:**

* `text`: The translated string (may contain `{placeholder}` variables)
* `source_hash`: SHA-256 prefix of the English source text. When the English
  text changes, this hash becomes stale, flagging the translation for review.

## Using Translations in Code

The public API is in `bridge/i18n/__init__.py`:

```python
from i18n import t, get_supported_languages, is_supported

# Basic translation
message = t("welcome", lang="de")
# "Willkommen bei AXOLENT AI!"

# With placeholders
message = t("rate_limit_warning", lang="en", remaining=30, window="hour")
# "You have 30 requests remaining this hour."

# Check language support
is_supported("fr")  # True
is_supported("xx")  # False

# List all languages
languages = get_supported_languages()
# [{"code": "en", "native": "English", "english": "English", "rtl": False}, ...]
```

**Rules:**

* Always use `t(key, lang)` for user-facing text. Never hardcode strings.
* The `lang` parameter comes from `LanguageContext.code` (resolved by
  `LanguageResolver`).
* If a key is missing in the requested language, `t()` falls back to English.

## Adding New Keys

### Step 1: Add to English Locale

Add the new key to `bridge/i18n/locales/en.json`:

```json
{
  "my_new_feature_title": {
    "text": "My New Feature",
    "source_hash": "PENDING"
  }
}
```

### Step 2: Sync to All Locales

```bash
python scripts/i18n_sync.py
```

This copies the key to all 20 locale files with `"source_hash": "PENDING"`
and the English text as placeholder.

### Step 3: Bootstrap Hashes

```bash
python scripts/i18n_bootstrap_hashes.py
```

This computes the SHA-256 hash of the English text and writes it to all
locale files.

### Step 4: Translate (Optional)

For non-English locales, replace the English placeholder text with the
actual translation. Community translations are welcome via PR.

**Note:** Initial translations are LLM-generated and may not be reviewed
by native speakers. The `source_hash` system ensures that when the English
source changes, stale translations are flagged.

### Step 5: Verify

```bash
# Run the i18n pre-commit hooks
pre-commit run i18n-check --all-files
pre-commit run i18n-scan --all-files
```

## Source Hash System

The `source_hash` field enables stale translation detection:

1. When a new key is added, `source_hash` is set to `"PENDING"`.
2. Running `i18n_bootstrap_hashes.py` computes `sha256(english_text)[:16]`
   and writes it to all locale files.
3. When the English text changes, the hash in `en.json` is updated.
4. The `i18n_check.py` pre-commit hook compares each locale's `source_hash`
   against the current English text hash. A mismatch means the translation
   is stale and needs review.

## Pre-Commit Hooks

### `i18n-check` (Parity Gate)

**Script:** `scripts/i18n_check.py`

**Checks:**

1. All locale files have exactly the same keys as `en.json` (no missing, no extra)
2. No `source_hash` fields are `"PENDING"` (must run bootstrap first)
3. `source_hash` matches current English text (detects stale translations)
4. All `{placeholder}` names in translated text match the English source
5. No empty `text` fields

**Runs on:** Changes to `bridge/i18n/locales/*.json`

### `i18n-scan` (AST Scanner)

**Script:** `scripts/i18n_scan.py`

**What it does:** Walks the Python AST of `presentation/` and `application/`
files to detect string literals passed directly to Telegram API methods
(e.g., `reply_text("Hello")` instead of `reply_text(t("greeting", lang))`).

**Watched calls:**

* `reply_text(...)`
* `edit_message_text(...)`
* `answer(text=...)`
* `send_message(...)`
* `InlineKeyboardButton(...)` (first argument or `text=` keyword)

**Whitelist rules:**

* Variable references are always OK (not string literals)
* `t()` calls or wrappers are OK
* Pure ASCII symbols (`"..."`, `"---"`) are OK
* Lines ending with `# i18n: ok` suppress the finding

**Runs on:** Changes to `bridge/presentation/*.py` and `bridge/application/*.py`

### `en-only-production`

**Script:** `scripts/check_en_only_production.py`

**What it does:** Blocks new German strings and comments in production code
paths. Enforces the English-only policy in all layers.

**Runs on:** Changes to `bridge/application/`, `bridge/domain/`,
`bridge/infrastructure/`, `bridge/presentation/`, `bridge/main.py`,
`scripts/`, `.pre-commit-config.yaml`

## Language Resolution Flow

The i18n system is tightly integrated with language resolution. The flow:

```
User sends message
  |
  v
LanguageResolver.resolve()
  |
  | Priority:
  |   1. /lang command override          -> confidence 1.0
  |   2. Sticky language (per chat_id)   -> confidence 1.0
  |   3. Auto-detection (domain/language) -> confidence varies
  |   4. Default fallback ("de")         -> confidence 1.0
  |
  | Smart-switch: auto-detection must exceed 0.7 confidence
  | to change the sticky language.
  |
  v
LanguageContext(code="de", source="sticky", ...)
  |
  v
t("welcome", lang=ctx.code)
  |
  v
"Willkommen bei AXOLENT AI!"
```

**Sticky behavior:** Once a language is set (via detection or /lang command),
it persists per `chat_id`. The user does not need to set language on every
message.

**Detection strategy** (in `domain/language.py`):

1. Non-Latin scripts detected via Unicode ranges (deterministic)
2. Cyrillic further classified as Russian vs Ukrainian
3. Latin-script languages scored by marker-word frequency
4. Shared diacritical characters resolved by word-marker precedence
   (see [ADR-0004](adr/0004-language-resolution-contract.md))
